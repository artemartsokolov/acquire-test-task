# Eqaring — Обоснование дизайн-решений

## Технологический стек

| Слой | Выбор | Обоснование |
|------|-------|-------------|
| **Сборщик** | Vite | Нулевая конфигурация, мгновенный HMR. Для прототипа приоритет — скорость итераций, а не продакшн-оптимизация. Vite даёт пересборку за доли секунды без сложности webpack. |
| **Фреймворк** | Vanilla JS (без React/Vue) | Осознанный выбор. Задание оценивает продуктовое мышление и понимание домена, а не знание фреймворков. Vanilla JS убирает все слои абстракции — код читается любым ревьюером без необходимости разбираться в JSX-компиляции, Virtual DOM или паттернах управления состоянием. Каждая строка аудируема. |
| **Стили** | Vanilla CSS с CSS custom properties | Дизайн-токены (цвета, отступы, радиусы, шрифты) определены как CSS-переменные в `:root`, что позволяет глобальную кастомизацию темы. Без Tailwind — потому что утилитарные классы делают HTML нечитаемым в кодовой базе, где HTML *и есть* конечный продукт. |
| **Шрифты** | Roboto (UI) + Roboto Mono (данные) | Индустриальный стандарт для B2B финтех-дашбордов. Roboto обеспечивает отличную читаемость на малых размерах (11–14px), что критично для информационно-насыщенных интерфейсов. Моноширинный вариант гарантирует табличное выравнивание чисел в финансовых таблицах. |
| **Роутинг** | На основе хэшей (`#/path`) | Одностраничное приложение без серверного роутинга. Хэш-роутинг работает без какой-либо серверной конфигурации, что позволяет запустить прототип мгновенно через `npm run dev` или даже по протоколу `file://`. |

### Почему не React/Next.js?

Компонентный фреймворк добавил бы ~200 КБ рантайма, сложность `node_modules` и когнитивную нагрузку для ревьюера. Всё приложение — ~1 200 строк JS и ~2 300 строк CSS. На этом масштабе чистый код **более поддерживаем**, а не менее. Ревьюер может найти любой элемент через `Cmd+F` мгновенно.

### Формат поставки

Прототип построен на чистом HTML/CSS/JS для максимальной портируемости и нулевого порога входа. Однако я могу предоставить тот же дизайн в **любом формате, который удобен команде**:

- **React / Next.js** — компонентная реализация с управлением состоянием, готовая к интеграции в существующую React-кодовую базу
- **Vue / Nuxt** — аналогичный подход в экосистеме Vue
- **Angular** — если стек команды этого требует
- **Figma-файлы** — полная дизайн-система с компонентами, токенами, auto-layout и интерактивными прототипами — готовая для передачи любой фронтенд-команде

Дизайн-решения, описанные в этом документе, **не зависят от фреймворка**. Структуры лейаутов, система отступов, цветовые токены и паттерны компонентов переносятся в любой стек напрямую.

## Дизайн-философия

### Эстетика «Институционального финтеха»

Визуальный язык намеренно монохромный и сдержанный — мы условно называем это **Swiss Bank-эстетикой**. Это не потребительский продукт, где важны «вау-эффект» и индивидуальность. Это B2B-инструмент для операционной команды, где:

- **Доверие** исходит от визуальной сдержанности, а не от визуального возбуждения
- **Сканируемость** важнее эстетики — операторы смотрят на это 8 часов в день
- **Плотность информации** должна быть высокой, но без ощущения перегруженности

Палитра намеренно ограничена:
- **Основной текст**: `#111827` (почти чёрный, не чисто чёрный — снижает нагрузку на глаза)
- **Акцент**: `#4F46E5` (индиго — профессиональный, не игривый)
- **Фоны**: `#FFFFFF` → `#F9FAFB` → `#F3F4F6` (трёхуровневая система глубины)
- **Опасность/Предупреждение**: Приглушённые красные и янтарные, никогда насыщенные — сигналы должны информировать, а не вызывать панику

## Что откуда взято

### От Stripe

Дашборд Stripe — золотой стандарт для UI платёжной инфраструктуры. Следующие паттерны напрямую заимствованы из Stripe:

| Элемент | Паттерн Stripe | Наша реализация |
|---------|---------------|-----------------|
| **Боковая навигация** | Левая панель с иконкой + лейблом, сгруппированная по бизнес-доменам (Payments, Balances и т.д.) | Идентичная структура: Overview, Disputes, Risk & Fraud, Merchants, Customers → группировка по ролям |
| **Песочница-баннер** | Жёлтая/янтарная полоса сверху: «You're testing in a sandbox» | Принято дословно — это индустриальная конвенция, пользователи её ожидают |
| **Лейаут деталей транзакции** | Две колонки: левая = хронологические данные (таймлайн, события, логи), правая = карточки сущностей (клиент, риск) | Тот же лейаут. Левая колонка рассказывает историю (что произошло), правая — даёт контекст (кто, уровень риска) |
| **Таблица Payment Breakdown** | Key-value строки с пунктирными разделителями, вычеты красным, нетто жирным | Практически идентично — так Stripe, Adyen и Checkout.com представляют разбивку комиссий |
| **Статус-бейджи** | Пилюлеобразные, цветокодированные (зелёный/красный/жёлтый/серый), с инлайн-иконкой | Тот же подход. `Succeeded ✓`, `Failed ✕`, `Disputed ⚠` |
| **Чипы карточных брендов** | Цветной фон с аббревиатурой бренда (VISA, MC, AMEX) | Прямое заимствование — мгновенно узнаваемый микро-паттерн |
| **Отображение API-ключей** | Обрезанный ключ с возможностью копирования | Упрощённая версия в карточке Recommendations |
| **Лог событий** | Обратно-хронологический список с таймстемпами | Разделён на «Recent activity» (для пользователя) и «Logs» (API-вызовы для разработчика) |

**Почему Stripe?** Потому что целевой пользователь (операционный менеджер хай-риск эквайрера) скорее всего уже использует Stripe как ориентир. Знакомость снижает когнитивную нагрузку. Отклонение от паттернов Stripe требует веской причины.

### От Paddle

Дашборд Paddle — более новый (редизайн 2024 года) и привносит современные B2B SaaS-конвенции, которые Stripe пока не принял:

| Элемент | Паттерн Paddle | Наша реализация |
|---------|---------------|-----------------|
| **Секция «Сегодня» с валовым объёмом** | Paddle открывается с KPI «Revenue» + живой таймстемп | Наша секция «Today» с `Gross volume` и `Yesterday` как двумя главными метриками с индикатором реального времени |
| **Филтр-пиллы выбора периода** | `Period: Last 7 days (Mar 26 – Apr 01)` + `vs: None` в виде пилюль-кнопок | Принято как фильтр-пиллы: `Last 7 days`, `Daily`, `Compare`, `Previous period` |
| **Карточки метрик с мини-графиками** | Revenue + Paid Transactions как парные карточки со спарклайн-графиками | Сетка из 6 карточек: Payments (донат), Processing Volume, Net Volume, Chargebacks, Failed & Declined, Top Merchants |
| **Таймстемп «По состоянию на»** | Каждая метрика показывает `As of 10:12 AM (UTC)` | Упрощено до `Updated 10 min ago` — менее точно, но более человечно |
| **Чистый плоский дизайн карточек** | Без теней, минимальные бордеры, белые карточки на слегка серому фоне | Карточки метрик используют `border: 1px solid var(--border)` на белом, без box-shadow |

**Почему Paddle?** Дизайн Paddle 2024 года более современный, чем у Stripe (который принципиально не менялся с 2020). Паттерн выбора периода и парные карточки метрик лучше подходят для дашборда, который должен отвечать на вопрос «Как дела у бизнеса прямо сейчас?»

### Оригинальное / Специфичное для домена

Эти элементы существуют потому, что **хай-риск эквайринг имеет уникальные требования**, которые ни Stripe, ни Paddle не решают (они обслуживают low-risk мерчантов):

| Элемент | Обоснование |
|---------|-------------|
| **Rolling Reserve (10%) в Payment Breakdown** | Хай-риск эквайреры удерживают 5–10% каждой транзакции в роллинговом резерве на 6 месяцев. Это вопрос №1 от мерчантов: «Где мои деньги?» Показав это инлайн с вычетами комиссий, мы делаем денежный поток прозрачным. Ни Stripe, ни Paddle этого не показывают, потому что не держат резервы. |
| **Коэффициент чарджбэков с порогом** | `0.0% ratio · Threshold: 1.0%` — программы Visa VDMP и Mastercard ECM штрафуют мерчантов выше 1% чарджбэк-рейта. Показав порог инлайн, мы даём оператору мгновенную проверку здоровья. |
| **Бар Risk Score** | Шкала 0–100 с цветовым градиентом (зелёный → янтарный → красный). Оценка риска на уровне транзакции — стандарт в хай-риск эквайринге (Nuvei, Worldpay, Checkout.com — у всех есть). |
| **Совпадение IP / страны карты** | Обнаружение кросс-граничного фрода. Если карта выпущена в Испании, но IP из Нигерии — это красный флаг. Stripe показывает это глубоко в UI Radar; мы выносим в детали транзакции, потому что в хай-риске каждая транзакция под лупой. |
| **Lifetime Volume + история чарджбэков клиента** | Клиент с €2 847 объёма и 0 чарджбэков — доверенный. Клиент с €50 и 2 чарджбэками — серийный абьюзер. Эта метрика меняет подход оператора к диспутам. |
| **Отображение версии 3D Secure** | `v2.2.0` — PSD2 требует SCA. Показав точную версию 3DS, мы помогаем compliance-офицерам верифицировать, что интеграция мерчанта соответствует регуляторным требованиям. |
| **Merchant Compliance Checklist** | Плавающая карточка с шагами онбординга (Profile → AML → Website Audit → Production Keys). Онбординг хай-риск мерчанта занимает 2–6 недель с множеством compliance-гейтов. Чеклист зеркалит реальный рабочий процесс. |
| **Device fingerprint + IP** | Стандартные данные для расследования фрода. Когда приходит чарджбэк, первое что проверяет аналитик: «Тот же устройство/IP, что и в предыдущих транзакциях этого клиента?» |

## Решения по информационной архитектуре

### Дашборд: двухуровневый лейаут

```
┌─────────────────────────────────────────────────┐
│  СЕГОДНЯ                                        │
│  ┌──────────────────┐  ┌──────────────────────┐ │
│  │ Валовый объём     │  │ Рекомендации         │ │
│  │ Вчера            │  │ API-ключи            │ │
│  │ [График]         │  │                      │ │
│  └──────────────────┘  └──────────────────────┘ │
│  [Баланс]  [CTA-баннер]                        │
├─────────────────────────────────────────────────┤
│  ОБЗОР                                          │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐    │
│  │Пл. │ │Об. │ │Нет │ │CBK │ │Отк.│ │Топ │    │
│  └────┘ └────┘ └────┘ └────┘ └────┘ └────┘    │
├─────────────────────────────────────────────────┤
│  ПОСЛЕДНИЕ ТРАНЗАКЦИИ                           │
│  [Таблица на 5 строк → «Все транзакции»]        │
└─────────────────────────────────────────────────┘
```

**«Сегодня»** = операционный вид в реальном времени (Что происходит сейчас?)  
**«Обзор»** = аналитический вид (Какой тренд?)  
**«Последние транзакции»** = вид для действий (Что требует внимания?)

Это зеркалит то, как операционный менеджер реально работает: взглянуть на сегодняшние цифры → проверить тренды → углубиться в отдельные транзакции.

### Детали транзакции: левая/правая колонки

```
┌──────────────────────────┬────────────────────┐
│ Детали транзакции        │ Оплачено           │
│ Разбивка платежа         │ €99.00             │
├──────────────────────────┤ Итого заработано   │
│ Последние действия       │ €76.37             │
│ (таймлайн)               ├────────────────────┤
├──────────────────────────┤ Риск и комплаенс   │
│ Метод оплаты             │ (карточка)         │
│ (двухколоночная сетка)   ├────────────────────┤
├──────────────────────────┤ Клиент             │
│ События                  │ (карточка)         │
│ (структурированный лог)  ├────────────────────┤
├──────────────────────────┤ Метаданные         │
│ Логи                     │ (пустое состояние) │
│ (API-вызовы)             │                    │
└──────────────────────────┴────────────────────┘
```

**Левая колонка** = хронологический нарратив (что произошло, в каком порядке)  
**Правая колонка** = контекст сущностей (кто, откуда, насколько рискованно)

Этот лейаут позволяет оператору читать левую колонку сверху вниз, чтобы понять жизненный цикл транзакции, а затем взглянуть вправо для контекста, когда что-то выглядит подозрительно.

### Правая колонка: карточный дизайн

Секции «Риск и комплаенс», «Клиент» и «Метаданные» используют светло-серые карточки (`#F9FAFB`) вместо ограниченных рамкой секций. Это создаёт визуальную группировку без добавления линий, которые конкурируют с информационно-насыщенной левой колонкой. Секция «Метаданные» использует пунктирную рамку (пустое состояние) для сигнализации возможности редактирования.

## Типографическая шкала

| Элемент | Размер | Начертание | Шрифт |
|---------|--------|------------|-------|
| Заголовок страницы (h1) | 24px | 700 | Roboto |
| Заголовок секции (h3) | 16px | 700 | Roboto |
| Основной текст | 14px | 400 | Roboto |
| Мелкие лейблы | 13px | 400 | Roboto |
| Моно-значения (ID, суммы) | 13px | 600 | Roboto Mono |
| Статус-бейджи | 12px | 600 | Roboto |

Шкала намеренно плотная (диапазон 12–24px). Финансовые дашборды должны вмещать максимум данных в минимуме пространства, оставаясь читаемыми. Каждый пиксель отступа — осознанный.

## Семантика цветов

| Цвет | Hex | Использование |
|------|-----|---------------|
| Индиго (акцент) | `#4F46E5` | Интерактивные элементы, ссылки, активные состояния |
| Зелёный (успех) | `#059669` | Успешные операции, пройденные проверки, положительные дельты |
| Красный (опасность) | `#DC2626` | Неуспешные, отклонённые, отрицательные суммы |
| Янтарный (предупреждение) | `#E69500` | Роллинговый резерв, диспуты, пороговые значения |
| Серый (нейтральный) | `#6B7280` | Вторичный текст, лейблы, неактивные состояния |

Цвета никогда не декоративны. Каждый цвет несёт семантическое значение, которое оператор запоминает один раз и распознаёт мгновенно.
